# LinkedIn Easy Apply Automation

An AI-powered LinkedIn Easy Apply bot. It browses job listings, scores them against your profile using an LLM, and fills out every Easy Apply form automatically — text fields, dropdowns, radios, checkboxes — all driven by your `profile.json` and a Groq LLM.

---

## How It Works

```
prefs.json  ──► search LinkedIn ──► score jobs (LLM)
                      │
                      ▼
              open Easy Apply modal
                      │
                      ▼
           handleFormStep() for each step
           ├── text/textarea:  resolveAnswer() first → LLM fallback
           ├── dropdown/radio: LLM first → resolveAnswer() fallback
           └── checkbox:       auto-check attestations → LLM otherwise
                      │
                      ▼
              Submit → log to data/linkedin-applications.json
```

All personal answers come exclusively from `data/profile.json`.  
All job search parameters come exclusively from `data/prefs.json`.  
No personal data is hardcoded in the script.

## Setup Steps

### 1. Clone the repo

```bash
git clone https://github.com/kpcreative/linkedin_easy_apply.git
cd linkedin_easy_apply
```

### 2. Install root dependencies

```bash
npm install
```

### 3. Install Playwright browsers

```bash
npx playwright install
```

### 4. Install web dashboard dependencies

```bash
cd web && npm install && cd ..
```

### 5. Create your `.env` file

```bash
cp .env.example .env
```

Open `.env` and replace the placeholder with your real Groq API key:

```
GROQ_API_KEY=gsk_your_actual_key_here
LLM_MODEL=llama-3.3-70b-versatile
```

Get a free key at [console.groq.com](https://console.groq.com) → API Keys → Create API Key.

### 6. Create your profile

```bash
cp data/profile.example.json data/profile.json
```

**Option A — Web dashboard (recommended):**

```bash
cd web && npm run dev
```

Open [http://localhost:3000/onboarding](http://localhost:3000/onboarding) and complete the onboarding wizard. It uses AI to extract your profile from a pasted resume or uploaded PDF/DOCX, then saves `data/profile.json` automatically.

**Option B — Edit manually:**

Open `data/profile.json` and fill in all fields. See `data/profile.example.json` for the complete schema.

### 7. Set your job preferences

```bash
cp data/prefs.example.json data/prefs.json
```

Edit `data/prefs.json` with your target keywords, location, and max applications per run. Or set everything from the web dashboard at [http://localhost:3000](http://localhost:3000).

## Running the Automation

### Option A — Web dashboard (recommended)

Start the dashboard:

```bash
cd web && npm run dev
```

Open [http://localhost:3000](http://localhost:3000), click **Start Automation**. Live logs stream in the dashboard. Click **Stop** to halt the process.

### Option B — Script directly

**Step 1 — Launch Chrome with remote debugging** (fewer bot-detection issues):

```bash
# Windows
"C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222

# macOS
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222
```

**Step 2 — Log in to LinkedIn** in that Chrome window.

**Step 3 — Run the script** in a second terminal from the project root:

```bash
npm run script:linkedin
```

If Chrome debugging is not available, the script falls back to launching its own Playwright Chromium browser and will prompt you to log in manually.

> **Restart required after profile changes:** The script reads `profile.json` once at startup. If you change your profile, restart the script to pick up the new data.

| LinkedIn / GitHub URLs | `profile.json` → `linkedinUrl`, `githubUrl` | Edit profile |
| Notice period | `profile.json` → `noticePeriodDays` | Edit profile |
| Relocation willingness | `profile.json` → `relocation` | Edit profile |
| Gender, veteran, disability | `profile.json` → `genderIdentity`, `veteranStatus`, `disabilityStatus` | Edit profile |
| Job search keywords | `data/prefs.json` → `keywords` | Edit prefs or dashboard |

### Hardcoded — must edit the script if your profile differs

These are in `scripts/linkedin-easy-apply.ts` and do **not** update when you change `profile.json`:

#### 1. Eligibility filter — experience cap
**Location:** `shouldApply()` function, ~line 1287

```typescript
// Skips any job requiring 2+ years of experience
const expSkip = /\b([2-9]|[1-9]\d)\+?...years?/;
```

**Change if:** You have more experience. Example — to allow jobs requiring up to 3 years, change `[2-9]` to `[3-9]`.

#### 2. Eligibility filter — tech stack skip list
**Location:** `shouldApply()` function, ~line 1290

```typescript
const techSkip = /golang|rust|kotlin|swift|embedded\s*c|vlsi|verilog|abap|salesforce|blockchain.../;
```

**Change if:** You know Golang, Kotlin, etc. — remove those terms from the regex. Add any tech you don't know to skip those roles.

#### 3. Seniority filter
**Location:** `shouldApply()` function, ~line 1285

```typescript
const senioritySkip = /senior|sr\b|lead engineer|tech lead|principal|staff engineer|director.../;
```

**Change if:** You are a senior candidate — remove the terms you want to allow.

#### 4. Additional months of experience default
**Location:** `resolveAnswer()` function, ~line 562

```typescript
if (/additional\s*months|months.*experience/i.test(l)) return '0';
```

**Change if:** You want to report more than 0 additional months.

## How the LLM Answers Form Questions

For every field the bot encounters on a form step:

- **Dropdowns and radio buttons:** LLM is called **first** with the question and all available option texts. It picks the most appropriate option based on your profile. If the LLM fails or returns an invalid option, `resolveAnswer()` (fast regex pattern matching) is the fallback.

- **Text and textarea fields:** `resolveAnswer()` runs first (no API call — covers 30+ common questions like name, email, phone, salary, notice period). Only unrecognized questions go to the LLM.

- **Checkboxes:** Legal attestation boxes (agree, certify, consent, authorize, acknowledge) are auto-checked. All other checkboxes go to the LLM.

The LLM is `llama-3.3-70b-versatile` on Groq by default (configurable via `LLM_MODEL` in `.env`). Groq's free tier is sufficient for typical runs.

## Why LLM Instead of Pure Deterministic Answers

A natural question: if name, email, salary, and notice period all come from `profile.json`, why not just hardcode the rest too?

### The problem with a fully deterministic approach

LinkedIn Easy Apply forms are **not standardized**. Each employer writes their own questions. In a single day the bot might encounter:

| Employer A | Employer B | Employer C |
|---|---|---|
| "Years of React experience?" with options `[< 1 year, 1–3, 3–5, 5+]` | "React proficiency?" with options `[Beginner, Intermediate, Advanced, Expert]` | "How long have you worked with frontend frameworks?" with options `[Less than 1 yr, 1–2 yrs, 2–4 yrs, 4+ yrs]` |

These three questions mean the same thing but have completely different option texts. A rule engine would need a hand-written case for every variation. There are thousands of employers — that's not maintainable.

Similarly for open-ended text questions:
- "Why do you want to work at this company?"
- "Describe how your background makes you a fit for this role."
- "What interests you about this position?"

A hardcoded answer to any of these would be obviously generic and identical across every application.

### What the LLM actually solves

| Problem | Without LLM | With LLM |
|---|---|---|
| Dropdown options vary per employer | Would need regex for every possible phrasing | LLM reads the actual options and picks the best match against your profile |
| Open-ended "why this role?" questions | Same canned answer everywhere | LLM reads the job title + company + your projects and writes a tailored response |
| Unexpected question phrasing | Field left blank or wrong answer selected | LLM understands intent regardless of exact wording |
| Job relevance — should I even apply? | Apply to everything (wastes applications) | LLM scores 0–100 against your skills; skips low-match jobs |

### The hybrid design and why it matters

The script uses a **deliberate split**:

```
Stable, factual fields (email, phone, salary, city, notice period)
  → resolveAnswer() — regex pattern matching, zero API calls, instant

Dynamic fields (dropdowns, radios, open text, job scoring)
  → LLM — context-aware, handles any phrasing, tailored per job
```

Doing everything with the LLM would be slower and more expensive. Doing everything deterministically would leave most forms broken or filled with generic copy-paste answers. The hybrid gives you speed on the easy stuff and intelligence where it actually matters.

### Concretely: what breaks without the LLM

- Every dropdown that doesn't exactly match a hardcoded regex → left blank or wrong option selected → application fails validation
- Every "cover letter" / "why this role" textarea → blank → looks like a bot
- Job scoring → disabled → applies to every result regardless of fit, burning your daily application limit on irrelevant roles

## Common Issues

**"Easy Apply button not found"**  
LinkedIn may have updated their DOM. Check `screenshots/debug-no-easy-apply-btn.png`. The job is added to the retry queue automatically and retried next run.

**Score always shows 50/100 in logs**  
`GROQ_API_KEY` is missing or invalid. Check `.env` and verify the key at [console.groq.com](https://console.groq.com).

**Bot gets logged out mid-run**  
Use Chrome remote debugging (Step 1 above) — it reuses your real logged-in Chrome session and is far more stable than a fresh browser.

**"Automation already running" error**  
A previous run left a stale PID. Restart the Next.js server or delete `data/automation.pid` if it exists.

**LLM picks the wrong dropdown option**  
Add a specific pattern for that question in the `resolveAnswer()` function in `scripts/linkedin-easy-apply.ts`. The function is documented with inline comments for each pattern.

**Typeahead/city field causes timeout**  
The script uses `humanTypeWithAutocomplete()` for location fields, which waits for and clicks the first suggestion. If it still times out, check `data/profile.json` → `city` and make sure it matches a LinkedIn-recognized city name.
